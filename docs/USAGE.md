# LLM Wiki 利用ガイド

**人間が日々どう使うか**の運用ガイド。設計思想は [`ARCHITECTURE.md`](ARCHITECTURE.md)、規約全文は [`../skills/llm-wiki/references/conventions.md`](../skills/llm-wiki/references/conventions.md)、LLM 向けの操作手順は [`../skills/llm-wiki/references/operations.md`](../skills/llm-wiki/references/operations.md) を参照。ここはそれらの手前にある「使い手の視点での流れと置き場」を扱う。

---

## 1. 素材から知識への流れ（3段モデル）

知識は「未分類の素材」から「スコープの定まったページ」へ、ingest を境に状態が変わる。

| 段階 | 場所 | scope | 誰が決めるか |
|---|---|---|---|
| ① 素材（未分類） | **Wiki の外**（Downloads / Obsidian / 任意フォルダ） | まだ無い | — |
| ② 取り込み（ingest） | `/wiki-ingest <path-or-url> [topic]` | **ここで global / topics を確定** | LLM が提案 → ユーザ確認 |
| ③ 確定 | `<scope>/wiki/<種別>/<slug>.md` ＋ `<scope>/raw/<source_id>.md`（証跡） | 確定 | — |

**要点: `scope`（global か topics/&lt;topic&gt; か）は ingest の瞬間に与えられる属性。** だから、まだ取り込んでいない素材は構造上スコープを持てず、論理的に Wiki の中には置けない。素材の置き場が「外」なのはこのためである。

---

## 2. 素材はどこに置くか（pre-scope）

- **未処理の素材は Wiki の外に置く。** Downloads でも Obsidian の vault でも、好きなフォルダでよい。`/wiki-ingest <path>` でそのパスを指して取り込む。
- **専用の `inbox/` は作らない。** 外部フォルダがそのまま「スコープ未確定ゾーン」の役割を果たす。仕組みを増やさず、概念だけ成立している状態。
  - 将来「外部だと散らかる／溜めて一括取り込みしたい」となったら、その時に初めて `inbox/` を正式化すればよい（オンディスク形式を壊さない追加で間に合う）。今は不要。
- **PDF・Excel・画像などのバイナリ原本も Wiki の外に置いたまま。** Wiki が抱えるのは原本そのものではなく、取り込み時に抽出したテキストである（§4）。原本は `raw/sources.md` に `source_id` ＋パス ＋ハッシュで記録して指すだけ。

---

## 3. 取り込む（ingest）

```
/wiki-ingest <path-or-url> [topic]
```

- URL でもローカルファイルでもよい。URL は WebFetch、ローカルは Read で中身を取得する。
- **取り込みの中でスコープが決まる。** LLM が「どのトピックに属すか／横断的か／新トピックを作るべきか」を提案し、**ユーザが確認**してから確定する。既存に無理に押し込まず、収まらなければ新トピック作成を提案する。
- 取り込みは要約ページの生成だけでなく、関連ページへの相互リンク・index・log の更新まで一括で行う。
- 自分の頭の中の知識を起こしたいときは ingest ではなく `/wiki-new`（§5）。

---

## 4. raw（原典の証跡）の扱い

`raw/` は「要約が正しいかを後から検証し、モデル更新時に再蒸留し、リンク腐敗に備える」ための地に足のついた真実。**読むだけ・触らない**のが原則。利用者として知っておくべきは次の3点だけ。

- **入るのは抽出テキストのみ。** `raw/<source_id>.md`（Read / WebFetch で得たテキスト）。**生のバイナリ（PDF/Excel）は入れない。** raw は flat で、サブフォルダに分けない。
- **バイナリ原本は外部のまま、`sources.md` に `source_id` ＋パス ＋ハッシュで参照。** Wiki を軽量なテキストの累積物に保つための判断。
- **中身は雑でよい。** 整形せず抽出生テキストのまま置く。むしろ加工しないことが「原典の真実」として正しい。配置の規律（`sources.md` への記録と `source_id` 由来のファイル名）は LLM とスクリプトが守るので、利用者が気にする必要はない。

> raw は健全性チェック（`wiki-validate.sh`）の走査対象ではない。リンクグラフにも index にも載らない。つまり raw に何を置いても整合は壊れないが、追跡できるのは上記の規律を守ったものだけである。

## 5. 日々の使い分け

| やりたいこと | コマンド | 駆動 |
|---|---|---|
| 外部のソース（記事・論文・URL・ファイル）を知識化 | `/wiki-ingest <path-or-url> [topic]` | ソース駆動 |
| 自分の頭の中の知識をページに起こす | `/wiki-new <title>` | 記述駆動 |
| Wiki を参照して出典付きで答えてもらう | `/wiki-query <question>` | 参照 |
| 整理（リンク切れ・孤立・矛盾・陳腐化・データギャップ）と改善提案 | `/wiki-lint [scope]` | 整理 |
| ページ移動・改名、トピック改名、被リンク確認 | `/wiki-move` | 再編 |
| トピック／ページ種別の追加・一覧・削除 | `/wiki-topic <add\|list\|remove>` | 管理 |

加えて、フックにより**意識しなくても**質問時に関連知識が自動参照される（[`CLAUDE_CODE_INTEGRATION.md`](CLAUDE_CODE_INTEGRATION.md)）。明示的に呼ぶのは取り込み・整理・再編といった「書く・動かす」操作のとき。

---

## 6. 自分で直接 Wiki を編集したいとき

LLM だけでなく人間が直接編集してもよい。ただし**層で権限を分ける**。

- **コンテンツ層（ページ本文・frontmatter のメタ）— 自由に編集してよい。** ただの Markdown で整合に影響しない。`updated:` を直すと丁寧。
- **簿記層（index.md / log.md / リンク `[[..]]` / slug / 移動・改名）— スクリプト経由が原則。** 手で触ると参照グラフが壊れる。特に移動・改名は手で `mv` せず必ず `/wiki-move`。
- **編集したら `/wiki-lint` で整合を回復（reconcile）する。** index 未掲載・孤立・リンク切れ・frontmatter 欠落を検出し、スクリプト経由で直せる。

原則は「**コンテンツは人間も自由、簿記は lint で回収**」。

---

## 7. やってはいけないこと

- 取り込んでいない素材を `<scope>/wiki/` の中に手で置く（スコープが未確定なものを構造に入れない）。
- バイナリ原本を `raw/` に抱え込む（既定は外部参照。raw はテキスト証跡のみ）。
- ページを手で `mv` する／index・log を手書きする（リンクグラフが壊れる。必ずスクリプト経由＝`/wiki-move`・`/wiki-lint`）。
- slug を手で命名する（`wiki-slug.sh` が決定論生成する）。

迷ったら `/wiki-lint` を実行する。ドリフトを検出し、回復手順を示してくれる。
